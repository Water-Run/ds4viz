# ds4viz 开发交接文档

> 写给下一任维护者(LLM 或开发者). 阅读本文件后再动手, 可避免踩 90% 的坑.

## 0. 项目一句话

数据结构可视化教学平台. 用户用 Python / C 写一段调用 `ds4viz` 库的代码, 库在执行过程中记录操作序列并生成 `.toml` 中间表示(IR), 渲染器读取 `.toml` 交互式回放.

架构三段式: **代码 → `.toml` IR → 渲染**.

## 3. IR 契约 - 最重要的一条约束

**`.toml` 输出格式是跨语言/跨组件的硬契约**, 任何语言库、渲染器、文档必须保持一致. 修改前必读 [`document/IR定义.md`](./IR定义.md).

要点速记:

- 顶层块: `[meta]`, `[package]`, `[remarks]` (可选), `[[steps]]`, `[[result]]` 或 `[[error]]`
- `meta.lang` / `package.lang` 仅允许 `python`, `c` (其他已归档)
- `value` 字段仅支持 `int | float | str | bool` (与各语言 API 对齐)
- 每个 step 必须带 `phase` (阶段标签)
- 异常路径必须生成 `[error]` 块, 而不是空文件

## 4. 开发工作流

### 4.1 一键启动(首选)

```bash
cp docker/.env.example .env
./docker/manage.sh up        # 构建并启动 postgres + backend + frontend
./docker/manage.sh logs      # 跟踪日志
```

访问:

- 前端: <http://localhost:5173>
- 后端: <http://localhost:10000/docs> (FastAPI 自动生成的 Swagger)
- postgres: `localhost:5432` (用户 `ds4viz` / 密码 `ds4viz_dev_pwd`)

**热重载**: 后端和前端的源码通过 volume 挂载进容器, 修改 `library/*`、`server/release/src`、`render/vue/src` 后, 分别重启容器即可(`./docker/manage.sh restart backend`).

> Docker 沙箱自动降级为 `timeout` 命令 (不依赖 systemd). 生产环境的 systemd-run 资源限制需要在宿主机上手动配置.

### 4.2 本地手动开发

如果不想用 Docker, 需要本机装齐:

| 子项目 | 工具链 |
|--------|--------|
| `library/python` | Python ≥3.12, `pip install -e .[dev]`, 然后 `pytest` / `ruff` / `mypy` |
| `library/c` | C17 编译器 (gcc/clang), `make test` / `make release` / `make memcheck` |
| `render/vue` | Node ≥20.19 或 ≥22.12, pnpm, `pnpm dev` / `pnpm test:unit` / `pnpm type-check` |
| `server/release` | Python ≥3.12, PostgreSQL ≥18, 按 `server/release/README.md` 初始化数据库, 然后 `python src/main.py --test` |
| admin-repl | Lua 5.4 + luarocks: `pgmoon`, `luaossl`, `replxx` |

### 4.3 修改某个子项目

1. **改语言库**: 修改 `library/<lang>` → 跑 `pytest`(Python) / `make test`(C) → 如果改了 IR 输出格式, 同步更新 `document/IR定义.md` 与 `render/vue/src/utils/ir.ts`.
2. **改前端**: 修改 `render/vue/src` → `pnpm test:unit && pnpm lint && pnpm type-check`.
3. **改后端**: 修改 `server/release/src` → `pytest -v` → 注意: 后端 API 改了, 前端要同步.
4. **改 IR**: 严重操作, 必须:
   1. 更新 `document/IR定义.md`
   2. 更新所有语言库
   3. 更新 `render/vue/src/utils/ir.ts` 解析器
   4. 更新 `server/release/src/service/execution_service.py` 验证逻辑
   5. 更新 `library/python/test/intergration_test.py` 的 `VALID_KINDS` / `VALID_LANGS`

## 5. 关键设计决策

### 5.1 沙箱 (sandbox_service.py)

后端执行用户代码的安全机制. 两个层次:

1. **首选**: `systemd-run --user --scope --property=MemoryMax=…M` (生产环境, 硬资源限制)
2. **降级**: `timeout --signal=KILL <秒数> <命令>` (Docker/无 systemd 环境)

切换逻辑在 `_check_systemd_available()`, 检测 `DBUS_SESSION_BUS_ADDRESS` + `XDG_RUNTIME_DIR` + 实测 `systemd-run --user --scope -- true`.

### 5.2 C 库头文件传递

`CExecutor.prepare_files()` 会把配置中 `c_ds4viz_header_path` 指向的 `ds4viz.h` 拷贝到每个执行工作区. 这样用户代码不用关心头文件在哪, 编译时直接 `#include "ds4viz.h"` 即可.

Docker 模式下路径固定为 `/app/library/c/ds4viz.h`, 见 `server/release/src/config/docker.yml`.

### 5.3 IR 错误块

所有语言库在异常路径上必须生成 `[[error]]` 块 (而不是抛出/返回空). Python 用 `RuntimeError` + context manager 在 `__exit__` 阶段写盘. C 库也类似. 这是为了渲染器永远有东西可以显示.

### 5.4 API 语言枚举 (`SupportedLanguage` StrEnum)

后端 `server/release/src/model/execution.py` 是单一事实源. 前端 `render/vue/src/utils/ir.ts` 的 `ALLOWED_LANGS` 必须与之一致. **不要在两边各写一份常量**.

### 5.5 数据库迁移

目前没有正式的迁移工具. 表结构在 `server/release/sql/init.sql`, 模板数据在 `server/release/sql/templates.sql`. 改动表结构时:

1. 同步 `init.sql` (给新部署用)
2. 写一段 `ALTER TABLE` SQL 给老部署用, 放到 `server/release/sql/migrate/` 目录(目前还没建, 需要时建)

## 7. 常见任务

### 新增一种可视化结构 (如: hash table)

1. 在 `document/IR定义.md` 增加该结构的 step schema
2. 在 `library/python/ds4viz/` 实现 Python 类 (参照 `stack.py` 的 context-manager 模式)
3. 在 `library/c/ds4viz.h` 实现对应的 C API (`dv_hash_table_*`)
4. 在 `render/vue/src/utils/ir.ts` 的 `ALLOWED_KINDS` 加入新类型
5. 在 `render/vue/src/components/` 增加对应的可视化组件
6. 在 `library/python/test/` 加测试
7. 在 `server/release/sql/templates.sql` 增加若干示例模板

### 改一个 API

`library/python` 与 `library/c` 必须同步. 后端的 `SupportedLanguage` 不需要改. 前端的 `ALLOWED_LANGS` 不需要改 (改的是语言枚举, 不是数据结构). 视情况更新 `document/IR定义.md`.

### 调整沙箱限制

`server/release/src/config/<env>.yml` 里的 `sandbox:` 段. 改完不需要重启数据库, 重启 backend 即可.

### 加一个新依赖

- Python 后端: 改 `docker/Dockerfile.backend` (RUN pip install) + 在文档中说明
- Python 库: 改 `library/python/pyproject.toml`
- Vue: 改 `render/vue/package.json`
- 不要直接 `pip install` 或 `pnpm add` 在本地后提交, 会漏掉 Dockerfile/package.json

## 8. 测试与质量门禁

每个子项目有独立的测试:

```bash
# Python 库
cd library/python && pytest -v && ruff check . && mypy ds4viz

# C 库
cd library/c && make test && make analyze

# Vue
cd render/vue && pnpm test:unit && pnpm lint && pnpm type-check

# 后端
cd server/release && pytest -v
```

修改 PR 前**必须**跑相关子项目的全套命令. 见 `AGENTS.md` 的 "Tests to run for changes" 段.

## 9. 已知约束 / 容易踩的坑

- **Vue 依赖 monaco-editor**, 体积大, 第一次 `pnpm install` 慢 (约 1-2 分钟). 不要删 `node_modules` 重新装, 用 `pnpm install --frozen-lockfile` 复用 lockfile.
- **后端 sandbox 不会捕获子进程派生的孙子进程**, 如果用户的 C 代码里 `fork()` + `exec()`, 可能逃逸出 timeout. 生产环境必须用 systemd-run + cgroup.
- **C 库执行用 `gcc -std=c2x`**, 测试时确认宿主机 gcc 支持 c2x. 切换标准需改 `sandbox_service.py:CExecutor.get_run_command`.
- **`document/IR定义.md` 是中文文档**, 修改后**保持中文**, 方便项目作者审阅.
- **文件名/路径不要用中文**, 即便项目作者用中文. 跨平台路径兼容.
- **不要在 `server/release/src/config/*.yml` 里硬编码密码** — 生产环境用环境变量覆盖. 当前 docker.yml 用 `DS4VIZ_DOCKER_JWT_SECRET_CHANGE_ME` 提醒.
- **`.archived/` 仍然受 Git 追踪**, 不要清理.
